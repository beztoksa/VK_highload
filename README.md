# Проектирование высоконагруженного аналога почты Mail.ru
## 1. Тема и целевая аудитория
### 1.1 Описание сервиса
Почта Mail.ru — проект компании, входящей в экосистему VK,
является одним из крупнейших в русскоязычном сегменте
интернета сервисов бесплатной электронной почты,
предоставляющим почтовые ящики на доменах mail.ru, internet.ru, list.ru, bk.ru, inbox.ru и др.

### 1.2 Целевая аудитория
- Количество пользователей в месяц(**MAU**): 48.5 млн за второй квартал 2025 [^1]
- Количество пользователей в день(**DAU**): 17.4 млн за первый квартал 2024 [^2]
- Географическое положение: Россия

### 1.3 Ключевой функционал сервиса
- Отправка/прием писем
- Организация папок, фильтрация почтового ящика
- Вложения писем и файлы
- Полнотекстовый поиск в письмах

### 1.4 Ключевые продуктовые решения
- Все письма и вложения хранятся на сервере
- Асинхронная доставка писем
- Полнотекстовый поиск по всем письмам
- Aнтиспам и безопасность писем
- Push уведомления

## 3. Балансировка нагрузки
### 3.1 Функциональное разбиение по доменам 
| Доменное имя      | Протокол | Назначение                    |
|-------------------|----------|-------------------------------|
| **mail.ru**       | HTTPS    | браузерные интерфейс          |
| **api.mail.ru**   | HTTPS    | основный API                  |
| **imap.mail.ru**  | IMAP/POP3 | для сторонних клиентов        |
| **smtp.mail.ru**  | SMTP     | обмен почтой с другими сервисами |
| **static.mail.ru** | HTTPS    | Отдача статики                |
| **media.mail.ru** | HTTPS    | Отдача вложений               |

### 3.2 Обоснование расположения ДЦ
| Локация                          | Обслуживаемый регион       | Назначение                                                          |
|----------------------------------|----------------------------|---------------------------------------------------------------------|
| **Москва**                       | Центральная Россия         | Core сервер(API, работа с БД)                                       |
| **Новосбирск**                   | Урал,Сибирь, Дальний Восток | Core сервер(API, работа с БД)                                       |
| **Амстердам**                    | Европа                     | Для отправки внешних писем, возникает проблема синхранизации        |
| **Все города милионники России** | Вся Россия                 | CDN, Edge сервера(Отдача статики, вложений, легковесные вычисления) |

Обоснование выбора:
В Москве расположены все основные ДЦ и через нее проходит наибольшее кол-во трафика
Так как задержка между Москвой и Дальним Востоком является существенной, ДЦ в Новосибирске является обоснованным, но возникает проблема передачи данных между
ДЦ. Ее можно решить либо разделением на непересекающиеся множество данных, либо асинхронной read-only репликацией.

ДЦ в Амстердаме также является обоснованным для быстрой передачи писем в другие страны, но опять же возникает проблема консистентности данных.

Для уменьшения общий задержки и для более быстрой отдачи вложений и статики, а также для более быстрой работы DNS предлагается использовать ДЦ в городах миллионниках, 
но они не будут выполнять core функции и напрямую работать с БД, поэтому нет проблемы консистентности данных. 
### 3.3 Распределение запросов по ДЦ 
Основная нагрузка по отдачи вложений(вложения 80% данных почты[^8]) будет распределена между slave серверами.
Поэтому на основные ДЦ будет возложена нагрузка по API, загрузке, передачи писем.
ДЦ в Амстердаме будет выполнять функции передачи писем между внешними сервисами.
### 3.4 Схема балансировки
Авторитетные DNS сервера буду расположены в главных ДЦ, а в побочные ДЦ будут просто кешировать ответы главных.
Авторитетные DNS сервера будут обмениваться информацией о загрузке ДЦ, а также картой RTT пользовательских IP.
На основе этих данных будет использоваться комбинированные подход выбора сервера(Least load + latency-based)

BGP Anycast будет использоваться в основном для главных ДЦ и побочных(только API) для быстрой реакции на отказ одного из ДЦ.
Для отдачи статики и вложений использовать anycast нецелесообразно из-за разрыва длительных соединений и вероятности определения пакетов на разные ip.
